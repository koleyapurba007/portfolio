TUTORIAL · IoT · SELF-HOSTED 2026

Building an IoT Monitoring Dashboard with Raspberry Pi, Node-RED, InfluxDB, Grafana & Mosquitto

A complete, self-hosted IoT monitoring pipeline on a single Raspberry Pi — sensor data published over MQTT, reshaped in Node-RED, stored in InfluxDB, and visualized on live Grafana dashboards, with an optional path to secure internet exposure via auto-renewing SSL.

Raspberry PiNode-REDInfluxDBGrafanaMosquitto MQTT

Overview

This project sets up a complete, self-hosted IoT monitoring pipeline on a single Raspberry Pi. Sensor data is published over MQTT, ingested and reshaped by Node-RED, written to a time-series database (InfluxDB), and visualized on live Grafana dashboards — accessible over the LAN or, optionally, exposed securely to the internet with auto-renewing SSL certificates.

Architecture

MQTTMosquitto Broker
→
FLOWNode-RED
→
STOREInfluxDB
→
VIEWGrafana Dashboard

1. Flash the OS

Burn an SD card with Raspberry Pi OS Lite (64-bit), default user pi. Boot the Pi, SSH in, and update the package index:

sudo apt update
sudo apt upgrade
A full apt upgrade isn't strictly required for this setup — updating the package index (apt update) is enough to proceed.

2. Install Node-RED, InfluxDB & Grafana

Rather than installing each service manually, this setup uses a well-known community installer script (Scargill's Pi provisioning script), which walks through installing Node.js, Node-RED, InfluxDB, and Grafana interactively:

wget https://bitbucket.org/api/2.0/snippets/scargill/kAR5qG/master/files/script.sh
bash script.sh
  • Script walkthrough: tech.scargill.net/the-script
  • Video guide: YouTube — Andreas Spiess

During the interactive prompts, select Node-RED with Node.js and accompanying parameters when asked, then select Grafana in a later run. Node-RED default login: admin / admin — change this before exposing the service.

Fixing broken package sources (NO_PUBKEY error)

The Grafana/InfluxDB APT repositories referenced by older scripts sometimes go stale, producing a NO_PUBKEY GPG error during apt update. If this happens, re-add the correct signing key and repo source before re-running the install — see this reference for the fix pattern. Once resolved, re-run the script and select Grafana again.

3. Configure InfluxDB for LAN access

Edit the InfluxDB config to enable the HTTP API and confirm the bind port:

sudo nano /etc/influxdb/influxdb.conf

In the [http] section:

[http]
  enabled = true
  bind-address = ":8086"

Make the startup script executable and connect using the Pi's LAN IP:

chmod +x /usr/lib/influxdb/scripts/influxd-systemd-start.sh
influx -host 192.168.2.216

Create a database user:

CREATE USER "pi" WITH PASSWORD 'Lazot@2020' WITH ALL PRIVILEGES
SHOW USERS

Once Node-RED starts injecting data into the target database, verify it landed correctly:

SHOW MEASUREMENTS
SHOW FIELD KEYS

4. Install and check Mosquitto MQTT broker

sudo apt-get install mosquitto
sudo systemctl status mosquitto.service
sudo netstat -tulpn

netstat confirms Mosquitto is listening on its default port (1883) alongside InfluxDB (8086) and Node-RED (1880).

5. Connect Grafana to InfluxDB

Open Grafana at http://<pi-ip>:3000 and add InfluxDB as a data source. The database name must match the database Node-RED is writing to, and auth should use basic auth with the InfluxDB user/password created above (pi / your password). From there, build panels against the measurements and fields confirmed in Step 3.

6. Optional — automate SSL certificate renewal

If the dashboard sits behind a Bitnami NGINX instance and is exposed with a domain name, certificates can be renewed automatically via a scheduled Let's Encrypt (lego) script. Create the renewal script (replace DOMAIN and EMAIL-ADDRESS):

#!/bin/bash
sudo /opt/bitnami/ctlscript.sh stop nginx
sudo /opt/bitnami/letsencrypt/lego --tls --email="EMAIL-ADDRESS" --domains="DOMAIN" \
  --path="/opt/bitnami/letsencrypt" renew --days 90
sudo /opt/bitnami/ctlscript.sh start nginx

Make it executable and schedule it monthly via cron:

sudo chmod +x /opt/bitnami/letsencrypt/scripts/renew-certificate.sh
sudo crontab -e
# add:
0 0 1 * * /opt/bitnami/letsencrypt/scripts/renew-certificate.sh 2> /dev/null
For multiple domains, update the --domains argument in the script to include each additional domain.

Troubleshooting — resetting certificates

If certificate generation fails and you need to start over, this removes generated output, restores the previous certs, and restarts services:

rm -rf /opt/bitnami/letsencrypt
sudo mv /etc/gitlab/ssl/server.crt.old /etc/gitlab/ssl/server.crt
sudo mv /etc/gitlab/ssl/server.key.old /etc/gitlab/ssl/server.key
sudo mv /etc/gitlab/ssl/server.csr.old /etc/gitlab/ssl/server.csr
sudo /opt/bitnami/ctlscript.sh restart

If a cron job was created for renewal, open sudo crontab -e and delete the renewal line.

Result

A fully self-hosted IoT monitoring pipeline running on a single Raspberry Pi: MQTT ingestion via Mosquitto, flow-based data shaping in Node-RED, time-series storage in InfluxDB, and live dashboards in Grafana — with an optional path to securely expose it over the internet with auto-renewing SSL.

Key Takeaways

  • Community install scripts save significant setup time for the Node-RED/InfluxDB/Grafana combo on Raspberry Pi, but package repositories can go stale — always be ready to patch GPG keys.
  • Database naming consistency between Node-RED's output and the Grafana data source is the most common point of failure when dashboards show no data.
  • LAN-first, then secure: get the stack working locally over LAN IP first, then layer on SSL/domain exposure once the pipeline is verified end-to-end.
